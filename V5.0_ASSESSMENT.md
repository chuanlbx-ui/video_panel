## 子项1：砍掉Puppeteer用WebCodecs直出

### 当前渲染流水线分析

```
Python: build_html(config) → index.html
         ↓
npx hyperframes render → Chrome/Puppeteer截图每帧 → FFmpeg合成视频
         ↓ (4-8分钟/条, ~1.5GB/视频)
edge-tts配音 + FFmpeg音频合成
```

核心瓶颈：`npx hyperframes render` 底层调用Puppeteer + Chrome，每帧截图、单线程渲染。

### 5个方案详细评估

#### 方案A：Python html2canvas替代Puppeteer
| 维度 | 评估 |
|------|------|
| **可行性** | ❌ 不可行。html2canvas是浏览器端JS库，Python没有等价实现。Python的`imgkit`(wkhtmltoimage)不支持CSS动画/GSAP/WebGL |
| **改动量** | N/A |
| **性能** | wkhtmltoimage不支持CSS3动画+JS执行，无法替代 |
| **风险** | 无法渲染GSAP动画，只能截静态图 |
| **结论** | **否决** |

#### 方案B：Pyppeteer（Python版轻量Puppeteer）
| 维度 | 评估 |
|------|------|
| **可行性** | ✅ 可行。Pyppeteer是Puppeteer的Python移植，API几乎相同 |
| **改动量** | **中等**：替换`app.py`中调用`npx hyperframes render`的部分（~50行），写Pyppeteer脚本控制Chrome渲染每帧（~200行）。但需要重写截图逻辑 |
| **性能提升** | **0-10%**：底层仍然是Chromium渲染，瓶颈不变 |
| **风险** | 维护成本高、Pyppeteer更新慢、与现有`npx hyperframes`的lint/validate流程冲突 |
| **结论** | **不推荐**。带宽瓶颈不变，纯换语言无意义 |

#### 方案C：rembg+pillow直接渲染（跳过浏览器）
| 维度 | 评估 |
|------|------|
| **可行性** | ❌ **不可行**。当前视频依赖GSAP动画、CSS渐变、粒子特效、Google Fonts排版。rembg是去背景工具，pillow无法执行JS动画 |
| **改动量** | 需要完全重写渲染引擎（几千行），丢失所有现有动画/特效 |
| **风险** | 极高：现有13个模板的CSS+GSAP动画全部废除 |
| **结论** | **否决** |

#### 方案D：node-canvas在Node.js端渲染
| 维度 | 评估 |
|------|------|
| **可行性** | ⚠️ 部分可行，但需重写动画系统。node-canvas是Canvas2D，不支持CSS+GSAP |
| **改动量** | **巨大**（~3000行）：需要将GSAP动画翻译为node-canvas逐帧绘制 |
| **风险** | 极高：丢失CSS排版精度、粒子系统、字体渲染不同 |
| **结论** | **否决** |

#### 方案E：保持Puppeteer但优化性能（推荐）
| 维度 | 评估 |
|------|------|
| **可行性** | ✅ **最可行** |
| **改动量** | **小**：约50行 |
| **具体优化** | ① `--workers auto`利用多核（当前是单线程）② `-q draft`快速预览③ 增加`--gpu`硬件加速④ 并行渲染队列扩展MAX_CONCURRENT（2→4）⑤ 帧采样（30fps→24fps或关键帧采样）|
| **性能提升** | **预估40-60%**：workers 2→4 + GPU加速 + 帧率降低 |
| **风险** | 低：不改架构，加参数即可 |
| **注意** | Workers翻倍内存也翻倍（2GB→4GB），需确认服务器RAM |

### 子项1推荐方案

> **方案E：保持Puppeteer + 性能优化**
> - 改动量：~50行（主要在`render_video()`和`process_queue()`）
> - 预期提速：40-60%（从40-90秒降到25-60秒）
> - 风险：低
> - 替代方案E+（长远）：如果未来需要极致速度，可以调研 WebCodecs API 在服务端渲染，但那需要将现有 hyperframes CLI 换成 Node.js 原生方案，属于 V6.0 范围

**具体改动点：**
1. `build_from_config.py` 的 `main()` 中 render 命令增加 `--workers auto --gpu`
2. `hyperframes_app.py` 的 `MAX_CONCURRENT` 从 2 提升到 4
3. 预览模式帧率从 30fps 降到 10fps
4. 添加 Render 缓存机制（相同config直接返回已有结果）

## 子项2：全自动生成（一句话出片增强）

### 当前状态
- `/api/one-shot` 已实现：关键词匹配模板 → 规则生成文案 → 渲染
- 前端：一个按钮 + `prompt()` 输入框
- 匹配引擎：纯关键词 `TEMPLATE_KEYWORDS` 字典

### 改进方案

#### A：接入AI（LLM）解析用户意图

**可行性**：✅ 高。项目中已有规则文案引擎（_generate_copy系列函数），LLM只需做：
- 模板匹配（从关键词升级为语义理解）
- 参数提取（品牌名、描述、地址、电话等）
- 文案优化建议

**改动量**：

| 文件 | 改动行数 | 说明 |
|------|---------|------|
| `hyperframes_app.py` | ~80行 | 新增 `_llm_match_template()` 函数，调用现有LLM接口解析用户输入 |
| 新增 `llm_client.py` | ~60行 | LLM调用封装（OpenAI/DeepSeek兼容） |
| `templates/index_v4.html` | ~30行 | 前端UI改造 |

**AI调用成本估算**：
- 每次解析：~300 tokens（输入）+ ~200 tokens（输出）
- 使用DeepSeek API（假设）：0.5元/百万token ~ 0.00025元/次
- 日均1000次 → 0.25元/天 → 几乎不计

**推荐实现方案**：
```
用户输入 → LLM结构化输出 {
  "template_id": "food_promo",
  "brand": "老张火锅",
  "description": "做了十五年正宗重庆火锅",
  "phone": "13888888888",
  "address": "步行街88号"
} → 调用现有_generate_copy() → 渲染
```

#### B：前端UI增强

| 改动 | 具体内容 |
|------|---------|
| 输入框 | 从 `prompt()` 改为内嵌大输入框（类似ChatGPT输入框），首页占据1/3屏幕高度 |
| 实时预览 | 用户输入后，显示匹配模板 + 生成文案预览，允许微调 |
| "再试一次" | 不满意可重新生成文案（调LLM出不同版本） |
| 历史记录 | 最近3条一句话出片记录可一键重做 |

**改动量**：
- `templates/index_v4.html`：~80行（UI改造）
- `templates/index_v4.css`（内联style）：~40行

#### C：匹配后预览+微调

**改动量**：
- `hyperframes_app.py`：~30行（新增 `/api/one-shot-preview` 和 `/api/one-shot-apply` 端点）
- 前端：~60行（预览确认界面）

### 子项2推荐方案

> **三步走方案**
> 
> **Step 1（V5.0 基础版）：** LLM解析替代关键词匹配 + 前端输入框增强
> - 改动量：~170行（后端80 + 前端90）
> - 效果：从"关键词匹配"升级到"语义理解"，匹配准确率大幅提升
> - AI成本：几乎零成本
> 
> **Step 2（V5.0 完整版）：** 加预览+微调界面
> - 额外改动量：~90行
> - 效果：用户可确认/修改再生成，减少废片率
> 
> **Step 3（V5.1）：** 模板扩展 + 多语言支持
